## Thinking Frame
Technology exists to serve people. Every technical solution has a human on the other end — a developer calling this API at 2am during an incident, a user clicking this button under time pressure, an operator reading these logs while stressed. What is their actual experience? Not what we decided they need — what they actually need.

## Approach
1. Walk through the user journey step by step — every touchpoint, every possible moment of confusion
2. Find the moment where the user has to hold the most in their head at once — reduce that cognitive load
3. Find the moment where an error is possible and design for graceful, informative failure
4. Ask: what does the user want to accomplish, not what does the system want them to do?

## When Most Effective
API design, developer tools, error handling and messages, onboarding flows, CLI design, anything where the human experience of a technical system directly affects adoption or outcomes.

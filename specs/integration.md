# Integration Specification

## 1 Purpose

The Creative Loop must integrate seamlessly into normal Claude Code workflows. It should feel like a natural extension of Claude Code's capabilities, not a separate system with its own UX. This spec defines how the loop gets activated, configured, and how its output flows back into regular work.

## 2 Activation Methods

### 2.1 Explicit Activation (Slash Command)

The primary activation method. User types a command to enter creative mode:

```
/creative How should we restructure the authentication system?
```

This triggers the meta-controller to:
1. Generate a creative brief from the user's input + project context
2. Start the creative loop
3. Present results when done

### 2.2 Suggested Activation

Claude Code recognizes tasks that would benefit from creative exploration and suggests it:

```
User: I need to redesign the caching layer. It's become a mess.

Claude: This looks like an architectural decision with many valid approaches.
        Would you like me to run a creative exploration? This would generate
        diverse approaches from different angles before we commit to a direction.
        [Yes / No, just do it the standard way]
```

**Trigger heuristics** for suggesting creative mode:
- Words like "redesign", "rethink", "explore options", "what's the best way"
- Architecture-level changes (affecting 5+ files)
- The user explicitly mentions trade-offs or being unsure
- Debugging sessions that have stalled (>3 failed attempts)

### 2.3 Configured Activation

Projects can configure automatic creative mode in their CLAUDE.md or `.creative-loop/config.json`:

```json
{
  "auto_activate": {
    "enabled": true,
    "triggers": [
      {"pattern": "architecture", "min_exploration_budget": "medium"},
      {"pattern": "api design", "min_exploration_budget": "low"},
      {"pattern": "performance optimization", "min_exploration_budget": "medium"}
    ]
  }
}
```

## 3 Project Setup

### 3.1 Initialization

First-time setup for a project:

```
/creative init
```

This creates:
```
.creative-loop/
├── config.json         # Default configuration
├── sessions/           # Empty, ready for sessions
├── patterns/           # Empty pattern libraries
│   ├── successful.json
│   ├── failed.json
│   ├── combinations.json
│   └── persona_effectiveness.json
└── personas/
    ├── builtin/        # Default personas (read-only reference)
    └── custom/         # Project-specific personas (user edits)
```

### 3.3 CLAUDE.md Integration

Projects add creative loop awareness to CLAUDE.md:


## 4 Output Integration

### 4.1 Presentation to User

After the loop completes, results are presented in a structured format:

```markdown
## Creative Exploration: Authentication Redesign

### Top Proposals

**1. Token Rotation with Sliding Window** (Score: 0.82)
- Core: Replace static tokens with time-windowed rotating tokens...
- Feasibility: High — uses existing JWT infrastructure
- Risk: Low — backwards compatible with grace period
- [Generated by: First Principles | Validated by: Experiment]

**2. Capability-Based Access Tokens** (Score: 0.78)
- Core: Each token encodes specific capabilities...
- Feasibility: Medium — requires permission model refactor
- Risk: Medium — migration complexity
- [Generated by: Cross-Domain Analogist | Partially validated]

**3. Wild Card — Zero-Trust Session Mesh** (Score: 0.65)
- Core: Eliminate persistent sessions entirely...
- Feasibility: Low-Medium — significant rearchitecture
- Risk: High — but high novelty payoff
- [Generated by: Constraint Inverter | Not yet tested]

### Recommended Combinations
- Proposals 1+2 complement each other: rotation handles session security
  while capabilities handle authorization

### What I'd suggest: Start with Proposal 1 (lowest risk, highest feasibility)
and evaluate Proposal 2 as a follow-up enhancement.

Want me to proceed with implementing one of these, explore further,
or refine a specific proposal?
```

### 4.2 Transition to Implementation

When the user picks an approach:
1. The selected proposal becomes the implementation plan
2. Creative memory records the selection
3. Normal Claude Code execution takes over
4. The creative loop can be re-invoked if the implementation hits a wall

### 4.3 Partial Usage

Users don't have to run the full loop. Quick modes:

- `/creative quick <problem>` — Single iteration, 3 personas, no experiments. Fast divergent thinking.
- `/creative evaluate <idea>` — Skip generation, just evaluate an idea the user already has.
- `/creative experiment <idea>` — Skip generation and evaluation, just test an idea.

## 5 Implementation as a Claude Code Extension

### 5.1 Core Implementation
The creative loop is implemented as a set of:
1. **Skill file** — `~/.claude/skills/creative.md` — Makes `/creative` available in every Claude Code session on the machine
2. **Prompt templates** — Stored in `.creative-loop/personas/` and `.creative-loop/prompts/` and loaded at runtime
3. **Config** — `.creative-loop/config.json` per project
4. **Memory** — `.creative-loop/patterns/` and `.creative-loop/sessions/` per project

The skill file contains the full orchestration logic. Prompt templates are loaded from the project directory at runtime, so they can evolve independently of the skill itself.

### 5.2 No External Dependencies
The creative loop requires:
- Claude Code (any version with sub-agent support)
- File system access (standard)
- Bash access (standard)
- No external services, databases, or APIs

### 5.3 Global Installation (This Machine)

Copy the skill file to make `/creative` available everywhere:

```bash
# From the creative-loop project directory
cp creative.md ~/.claude/skills/creative.md
```

That's it. `/creative` now works in any Claude Code session on this machine.

### 5.4 Installation on Another Machine

Two steps:

**Step 1**: Copy the skill file
```bash
# Copy creative.md to ~/.claude/skills/ on the new machine
# Either manually or via the repo
```

**Step 2**: Init a project (when needed)
```
/creative init
```
This creates `.creative-loop/` in the current project with all templates and default config.

### 5.5 Keeping the Skill Updated

The skill file is in this repo. To update it on any machine:
```bash
cp creative.md ~/.claude/skills/creative.md
```

Or if Claude Code itself improves the skill during self-optimization (`/creative optimize`), it edits `~/.claude/skills/creative.md` directly.

### 5.6 Portability of Project Memory
The `.creative-loop/` directory is project-local and can be:
- Committed to version control (creative memory travels with the project)
- Shared across team members (they get the learned patterns for free)
- Backed up and restored
- Copied to seed related new projects with relevant patterns

### 5.7 Self-Modification
The system can improve itself. Claude Code can edit:
- `~/.claude/skills/creative.md` — Improve the orchestration skill
- `.creative-loop/personas/builtin/*.md` — Strengthen persona prompts
- `.creative-loop/prompts/*.md` — Refine phase instructions
- `.creative-loop/config.json` — Tune parameters

This is just file editing — the same thing Claude Code does for any project file. The result is a system that gets better at creative thinking through use.

See [specs/self-optimization.md](self-optimization.md) for details.
